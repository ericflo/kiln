Demo · 60 seconds · one GPU

Watch the loop close in 60 seconds.

Cold start. Ask the base model who Kiln is — it has no idea. POST a two-example correction to /v1/train/sft. The server trains the LoRA in-process, hot-swaps it at the next iteration boundary, and the very next chat completion uses the updated weights. No restarts, no second model in VRAM. One process, one GPU, one continuous shell session.

Recording note: this embedded asciicast was captured on kiln v0.2.8, so the startup banner shows 0.2.8. The current release line is kiln-v0.2.13; the SFT training, adapter hot-swap, and chat-completions API flow shown here remains current.

Scene 1 — start Kiln

The recording starts the server from the repo root with the Qwen3.5-4B weights and the example config. The banner reports GPU inference, the model path, VRAM, and the listen address before the server accepts requests. 

KILN_MODEL_PATH=./Qwen3.5-4B ./target/release/kiln serve --config kiln.example.toml
Model:   ./Qwen3.5-4B
CUDA:    available ✓
GPU:     NVIDIA RTX A6000
Listen:  http://127.0.0.1:8420

Scene 2 — ask the base model

The first chat completion asks what Kiln is before any adapter is trained. The base model answers about pottery, which is coherent but wrong for this project. 

curl -s http://localhost:8420/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"messages":[{"role":"user","content":"In one short sentence, what is the Kiln inference server?"}],"max_tokens":48,"temperature":0.0}' \
  | jq -r '.choices[0].message.content'
Kiln is a tool for managing kiln operations in pottery and ceramics, often used to control firing schedules and monitor temperatures.

Scenes 3–4 — train a correction

The demo submits two supervised correction examples to /v1/train/sft, then polls /v1/train/status until the demo adapter finishes training. 

curl -s http://localhost:8420/v1/train/sft \
  -H 'Content-Type: application/json' \
  -d @demo-sft.json \
  | jq -c '{job_id, state}'

curl -s http://localhost:8420/v1/train/status \
  | jq -c '.[-1] | {state, adapter_name, current_loss, elapsed_secs}'
{"job_id":"...","state":"pending"}
{"state":"completed","adapter_name":"demo","current_loss":0.0241,"elapsed_secs":62.4}

Scene 5 — ask with the trained adapter

The second chat completion uses the same prompt and explicitly selects the trained demo adapter. No restarts or second model process are involved. 

curl -s http://localhost:8420/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"adapter":"demo","messages":[{"role":"user","content":"In one short sentence, what is the Kiln inference server?"}],"max_tokens":80,"temperature":0.0}' \
  | jq -r '.choices[0].message.content'
Kiln is a single-GPU inference server for Qwen3.5-4B with live LoRA training over HTTP — submit corrections and the model improves in seconds.

Scene 6 — confirm the adapter exists

The final command lists LoRA adapters through /v1/adapters. The trained demo adapter is persisted on disk and available for future requests. 

curl -s http://localhost:8420/v1/adapters \
  | jq -c '.available[] | {name, size_bytes, modified_at}'
{"name":"demo","size_bytes":12582912,"modified_at":"2026-05-01T04:39:42.512Z"}

What you’re watching

The demo is a single uncut shell session. It runs the same prompt twice — once before a correction is submitted, once after — so the contrast is unmistakable. The whole thing is silent: there is no narration track, no jump cuts, no editing. What you see is what the kiln server actually does.

Scene by scene

Scenes 1–2. The kiln binary starts. The startup banner shows the model, the GPU, the VRAM budget, and the listen address. Then a chat completion: “In one short sentence, what is the Kiln inference server?” The base Qwen3.5-4B has never heard of Kiln-the-software, so it guesses pottery. A reasonable wrong answer.

Scenes 3–4. A single curl POST to /v1/train/sft with two correction examples and an adapter name. The server queues the job, training runs in-process on the GPU we’re already serving from, and a /v1/train/status poll shows the job completed in a few seconds. The server never paused.

Scenes 5–6. The same prompt as scene 2 — verbatim. The model now answers correctly with the language from the correction examples. A final /v1/adapters list confirms the new adapter is active, rank 8, about 4 MB on disk, ready for the next correction to stack on top.

Why this loop matters

Today, improving a deployed model is a multi-hour multi-process pipeline: collect failures, format them, upload to a training service, wait, download new weights, redeploy, hope. The cost of every iteration is high enough that nobody iterates — the model you ship is the model your users live with. Kiln collapses that into one HTTP call. The activation energy of a correction drops from hours to seconds, and the model you ship is the one that just learned from your last failure.

Read the setup path in the Quickstart and Architecture docs. If your local run differs from the recording, use the Troubleshooting page to check the binary, model path, /health, and adapter directory first.

Reproduce it yourself

The canonical demo runs the same commands you’d run as a first-time user. Every line is in the Quickstart. The recording protocol — terminal size, scene timing, expected outputs — is in SCRIPT.md.
