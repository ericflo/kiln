## Purpose

PR #644 implements PR #643 audit Path 1 remediation: it auto-gates
`fused_rmsnorm_with_autograd` on detected total VRAM with a 47 GiB
threshold (48 128 MiB). A6000 (49 140 MiB ≥ 48 128 MiB) routes to the
fused path; A40 (46 068 MiB < 48 128 MiB) routes to the candle-op
fallback. The gate is therefore a **no-op on A6000** — A6000's path under
#644 is identical to A6000's path under #638 alone.

This audit answers: **"does the post-#644 stack on A6000 close the
original Phase 10 §1 goal of T=8192 SFT fitting on A6000 (48 GiB) without
OOM and producing correct loss?"** Three possible outcomes per the
audit's task description:

1. **GREEN** — T=2048 and T=8192 both fit with finite loss → Phase 10 §1
   is closed; record the closure and move on.
2. **PARTIAL** — T=2048 fits but T=8192 OOMs → document the gap and
   the next remediation needed.
3. **RED** — T=2048 still OOMs → critical, recommend lowering the gate
   threshold or escalating to a deeper audit Path.

Per PR #642, A6000 with the trainer's auto-config seg=4 OOMs at T=2048
during segment-0 recompute at the same wall-clock peak (48 554 MiB) as
the candle-op fallback, so the audit standardizes on
`KILN_GRAD_CHECKPOINT_SEGMENTS=8` (PR #642's "ok at peak 43 178 MiB"
configuration). The seg=16 cells are added here to test whether more
aggressive checkpointing closes the T=4096/T=8192 gap.

## Failure modes

Two distinct OOM signatures appear at T=8192 on A6000:

**Mode A: ceiling-OOM** (peak 48 360–48 648 MiB across T=4096 seg=8/16,
T=8192 seg=8 ON, T=8192 seg=16 OFF). The candle CUDA caching allocator
fills its pool to the GPU ceiling and OOMs on the next allocation
attempt. Step time 3.6–6.4 s indicates failure during the first forward
pass or first backward recompute, before training step 1 completes.

**Mode B: low-peak OOM** (peak 39 048 MiB, T=8192 seg=16 ON, both clean-
and dirty-allocator runs). Step time 5.66–6.14 s — same range as Mode A.
Peak only reaches 39 GiB, then the bench classifies the result as OOM
because the candle error message contains `out of memory` /
`OutOfMemory` / `CUDA_ERROR_OUT_OF_MEMORY`. The 10 GiB headroom between
peak and ceiling implies the failure is **a single contiguous allocation
attempt for a tensor that does not fit in the available pool segment**,
not gradual VRAM exhaustion. Candidate sources for this single
allocation, ranked by likely size at T=8192:

1. **MLP gate-projection scratch.** Qwen3.5-4B has `ffn_dim≈18944`. A
   `[1, 8192, 18944]` bf16 intermediate is 311 MiB per layer; at the
   recompute peak with 2 layers per segment that is 622 MiB. Not
   single-allocation-fail-sized on its own, but a temporary that the
   pool needs a contiguous chunk for.
2. **F32 cross-entropy logits or backward grads** — even with FLCE on
   (which avoids materializing the full `[T, V]` tensor), per-chunk
   contiguous slices may be the failing single allocation. PR #631
   documented FLCE Phase A's chunked-vocab contig bug, fixed in PR #633;
   this audit runs against `130902a` which includes #633, but the
   single-chunk size at T=8192 may still exceed pool-fragmentation
   limits.
3. **GDN recurrent-state intermediates.** GDN's linear-attention forward
   has scratch tensors that scale with T per layer. The 24 GDN layers
   dominate Qwen3.5-4B's per-layer cost at long T (per the
   `kiln-fp8-kv-gdn-bottleneck` agent note: GDN is the memory bottleneck
   during prefill, not the GQA paged KV cache).
4. **Backward saved-tensor for FLCE final-projection chunk** — already
   reduced by PR #638's CustomOp2 forward pass, but the backward chunked
   matmul still needs a contiguous output buffer.

Distinguishing between these requires a per-allocation tracing patch
(or candle's CUDA allocator instrumentation) that this audit does not
add. Mode B is the strictly-better OOM signature — peak is 9.5 GiB
lower than the OFF arm at the same (T, seg) — but is still RED.

## Verdict

**PARTIAL.** PR #644's Path 1 gate landed cleanly and works as designed
on A6000. The T=2048 SFT regime on A6000 is fully closed (peak 42 GiB
at seg=8, finite loss, bit-exact between seg=8 and seg=16, bit-exact
between RMSNorm ON and OFF per PR #642's parity finding). The PR #638
saved-tensor reduction delivers its intended ~9.5 GiB peak savings on
A6000 at T=8192. **But the original Phase 10 §1 motivating goal of
"T=8192 SFT fits on A6000 (48 GiB)" is not achieved** — both T=4096
and T=8192 OOM under every (seg, RMSNorm op) configuration tested,
including the maximally-checkpointed seg=16 + CustomOp2 ON configuration.

The Phase 10 §1 stack (#637 layer-pair tiled + #638 CustomOp2 + #644
VRAM gate) should be considered **closed within its scope** —
specifically, RMSNorm fusion landed correctly and bit-exact; A40
regression mitigated; T=2048 SFT works on A6000. The remaining T=4096
and T=8192 SFT-on-A6000 problem is a **different bottleneck class**
that the RMSNorm-focused Path 1/2/3 menu from PR #643 does not address.

## Recommended next slice

**Do NOT queue a "Path 2 candle-allocator pool tuning" or "Path 3
revert kernel files of #638" task to address the A6000 T=8192 gap.**
PR #643's Path 2 and Path 3 were both proposed remediations for the
A40 regression mechanism (where binary-footprint changes perturbed the
candle allocator's pool sizing decision on A40-class GPUs). On A6000
the allocator already sits in the larger pool by default, so neither
Path 2 (tuning the pool decision) nor Path 3 (reverting #638's kernel
files) would change A6000's per-step VRAM profile materially. PR #642
proved this directly: A6000 was bit-identical between c983ca7 (pre-#638)
and 1edecd7 (post-#638) at T=2048. The candle-op fallback at T=8192
seg=16 in this audit (`OOM peak 48 584 MiB`) is further evidence —
even reverting the kernel entirely keeps T=8192 OOMing on A6000.

The A6000 T≥4096 gap is dominated by per-allocation peak (Mode A) and
single-contiguous-chunk fragmentation (Mode B), not by RMSNorm
saved-tensor cost. The right next slice is therefore Phase 10 §2 and §3
of the project roadmap:

1. **Phase 10 §2 — Liger RoPE.** Fused rotary-positional-embedding
   reduces per-layer transient memory traffic. At T=8192, RoPE is
   applied to `[1, T, num_heads, head_dim]` Q/K tensors twice per
   layer (Q and K). Fusing the elementwise sin/cos multiply into the
   RoPE kernel eliminates intermediate Q/K-rotated tensors that the
   candle chain materializes.
2. **Phase 10 §3 — Liger SwiGLU/GeGLU.** Fused gated activations target
   the `[1, T, ffn_dim]` MLP gate-projection intermediate (the
   311 MiB-per-layer scratch flagged in [`#failure-modes`](#failure-modes)).
   On Qwen3.5-4B with `ffn_dim≈18944`, fusing the SiLU/GELU gate-mul
   into the down-projection epilogue eliminates the largest single
   per-layer activation.
3. **Pre-§2/§3: per-allocation tracing patch on A6000 at T=8192 seg=16.**
   Mode B (peak 39 GiB + OOM) is unexplained at the per-allocation
   level. A short instrumentation patch logging every `cudaMalloc` /
   `cudarc::driver::DeviceRepr::alloc_zeros` call > 100 MiB would
   pinpoint the failing single allocation, which prioritizes whether
   §2 (RoPE) or §3 (SwiGLU) is the right first kernel to land for the
   T=8192 closure.

This audit recommends **proceeding directly to (3) — the per-allocation
tracing patch on A6000 at T=8192 seg=16 — as the next bench-only
no-PR-merge investigation**, then deciding §2 vs §3 based on which
single allocation Mode B is failing on. If neither (3) nor (2)/(3)
closes T=8192 within ~6 weeks of work, the next escalation is to drop
T=8192 from the Phase 10 §1 motivating goals and re-target it to a
larger-VRAM SKU (RTX 6000 Ada Generation 48 GiB shows the same OOM
signature as A6000; an RTX PRO 6000 Blackwell 96 GiB would fit but is
post-CUDA-13 compute capability — see kiln skill GPU compatibility
matrix).

The Phase 10 §1 RMSNorm work itself — including PR #637 (layer-pair
tiled), PR #638 (CustomOp2), and PR #644 (gate) — should be marked
**complete** in the project roadmap. The "T=8192 SFT on A6000" goal
should be moved from §1 to §2/§3.

## Bench harness

Three patcher scripts were used, each idempotent (detects the
`// PHASE10-CLOSURE-PATCH` marker and rewrites cleanly). Reproduced
here for reproducibility (the 6-cell `phase10_rmsnorm_bench.rs` on
main is **not** modified by this PR — patches are applied on demand):

### Multi-cell variant (gate verification + T=2048 + T=8192)

```python
# closure_bench_patch.py — replaces the 6-cell main() with T=128, T=2048,
# T=8192 (all custom_op=ON). Used for the gate-verification + main bench
# pass.
import pathlib, re, sys
BENCH = pathlib.Path("crates/kiln-server/examples/phase10_rmsnorm_bench.rs")
MARKER = "// PHASE10-CLOSURE-PATCH"
src = BENCH.read_text()
if MARKER in src:
    print(f"{BENCH}: already patched, no-op"); sys.exit(0)
new_block = f'''    {MARKER}
    rows.push(run_one(128, true, &tokenizer, &model_config, &gpu_weights, baseline_mib)?);
    rows.push(run_one(2048, true, &tokenizer, &model_config, &gpu_weights, baseline_mib)?);
    rows.push(run_one(8192, true, &tokenizer, &model_config, &gpu_weights, baseline_mib)?);
    print_table(&rows);
    Ok(())
}}
'''
m = re.search(r"    // Cells 1-2: T=2048.*?\n\}\n", src, re.DOTALL)
BENCH.write_text(src[:m.start()] + new_block + src[m.end():])
```
