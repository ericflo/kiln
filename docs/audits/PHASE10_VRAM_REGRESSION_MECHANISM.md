# Phase 10 §1 — VRAM regression mechanism (Tier 1 + Tier 2)

Date: 2026-04-29
Status: **Tier 1 closes hypothesis (3). Tier 2 is null on A6000 — the +18 GiB
regression observed on A40 in [`PHASE10_VRAM_REGRESSION_BISECT.md`](PHASE10_VRAM_REGRESSION_BISECT.md)
does not reproduce on A6000 with the same instrumented harness.**
Hardware: NVIDIA RTX A6000 (49 140 MiB, sm_86), CUDA 12.4, RunPod pool pod
`pod-cb7a166f8cd2d98f2f7466db` (kiln-runpod image)
Bench: slim variant of `crates/kiln-server/examples/phase10_rmsnorm_bench.rs`
(T=2048 + custom_op=true, single SFT cell after a T~256 warmup) — replaces the
6-cell bench so the harness fits inside the 90 min/$40 wall-clock cap. Bench
file is reproduced in [`#bench-harness`](#bench-harness) below.
Patches: Tier 1 dispatch tracing in `forward.rs::rms_norm()`; Tier 2
per-segment VRAM probes (gated by `KILN_PHASE10_VRAM_PROBE=1`) inside
`checkpointed_forward_backward()`. Both patches were generated by a single
Python tool (also reproduced below) so the same script runs at both commits.
Companion to: [`PHASE10_VRAM_REGRESSION_BISECT.md`](PHASE10_VRAM_REGRESSION_BISECT.md)
which pinned PR #638 as the regression-introducing PR but left the mechanism
unpinned. [`PHASE10_RMSNORM_E2E.md`](PHASE10_RMSNORM_E2E.md) is the original
e2e SFT validation that documented the regression but not the bisect.

## TL;DR

Two tier-results, one verdict on hypothesis (3), one **null** on hypotheses
(1) and (2):

1. **Tier 1 — kernel kill switch is fully honored.** With
   `KILN_DISABLE_RMSNORM_KERNEL=1` set on the command line at `1edecd7`,
   every `rms_norm()` dispatch site emits a `tracing::info!` line with
   `kernel_disabled=true, take_kernel=false`, then takes the
   `rms_norm_fallback` candle-op chain. Across the warmup pass +
   T=2048 main step at `seg=4` (auto-config) and `seg=8`
   (`KILN_GRAD_CHECKPOINT_SEGMENTS=8`), **zero** dispatch sites take the
   `fused_rmsnorm_with_autograd` kernel branch. The four call shapes
   observed all map to RMSNorm sites that are byte-equal between c983ca7
   and 1edecd7 (input_norm `[1, T, 2560]`, post_attn_norm `[1, T, 2560]`,
   q_norm `[1, T, 16, 256]`, k_norm `[1, T, 4, 256]`).
   **Hypothesis (3) (subtle dispatch bypass) is RULED OUT.**

2. **Tier 2 — per-segment VRAM walk is byte-identical between c983ca7 and
   1edecd7 on A6000 at both seg=4 and seg=8.** The +18 GiB regression
   pinned on A40 in PR #641 does not reproduce. Both arms produce the
   same probe sequence, the same peak VRAM, and the same final loss to
   the last digit (2.4473333… at seg=8). At seg=4 (auto-config) both
   arms OOM during segment-0 recompute at the same wall-clock peak
   (48 554 MiB); at seg=8 both arms succeed at peak 43 178 MiB.

3. The regression on A40 is therefore **GPU-specific** to A40 (or, more
   precisely, to whatever class of GPU configuration A40 represents that
   A6000 does not). The most parsimonious explanation is hypothesis (1)
   — the candle CUDA caching allocator's pool sizing depends on
   GPU-detected total VRAM, AND a binary-footprint change in PR #638
   nudges the A40 detection toward a fatter pool (one that A6000 already
   uses by default). This is **not** the same thing as confirming
   hypothesis (1); see [`#interpretation`](#interpretation).

## Tier 1 — dispatch trace

Patch (in `crates/kiln-model/src/forward.rs`): a `tracing::info!` line at the
top of `rms_norm()`'s `#[cfg(feature = "cuda")]` block, logging
`kernel_disabled`, `bwd_disabled` (only at 1edecd7), `supports`, the resulting
`take_kernel` boolean, and the input shape. The branch decision is computed
once and the log fires before either the kernel call or the fallback.

Run (1edecd7, A6000, kernel kill switch on, seg=4 auto-config — the most
realistic dispatch coverage; the warmup completes successfully and emits
hundreds of dispatch lines before the main cell OOMs):

```
KILN_DISABLE_RMSNORM_KERNEL=1 KILN_PHASE10_VRAM_PROBE=1 \
RUST_LOG=kiln::rms_norm_dispatch=info,kiln=warn,warn \
  ./target/release/examples/phase10_rmsnorm_bench --model-path /workspace/qwen3.5-4b
```

Sample log lines (representative — full log captured in the bench
session has **445 dispatch lines** for the seg=4 (auto-config) run at
1edecd7, **all 445 with `take_kernel=false`**):

```
INFO kiln::rms_norm_dispatch: rms_norm dispatch kernel_disabled=true bwd_disabled=false supports=true take_kernel=false shape=[1, 256, 2560]
INFO kiln::rms_norm_dispatch: rms_norm dispatch kernel_disabled=true bwd_disabled=false supports=true take_kernel=false shape=[1, 256, 16, 256]
INFO kiln::rms_norm_dispatch: rms_norm dispatch kernel_disabled=true bwd_disabled=false supports=true take_kernel=false shape=[1, 256, 4, 256]
…
INFO kiln::rms_norm_dispatch: rms_norm dispatch kernel_disabled=true bwd_disabled=false supports=true take_kernel=false shape=[1, 2048, 2560]
```

**Verdict.** Every observed line has `take_kernel=false`. The four shape
classes — `[1, T, 2560]` (pre/post-attn norm and final norm),
`[1, T, 16, 256]` (q_norm), `[1, T, 4, 256]` (k_norm), and the
gated-DeltaNet variants — all dispatch through `rms_norm_fallback` exactly as
intended. Hypothesis (3) is ruled out: there is no subtle correctness bug
where a dispatch site bypasses the kill switch. The same set of sites is
exercised at c983ca7 (which only has the `kernel_disabled` env var, not
`bwd_disabled`); the log at c983ca7 is structurally identical with one fewer
field and confirms the same `take_kernel=false` at every site.

## Tier 2 — per-segment VRAM walk

Patches (in `crates/kiln-train/src/trainer.rs`):

* `_phase10_vram_mib()` shells to `nvidia-smi
  --query-gpu=memory.used --format=csv,noheader,nounits` and parses the
  current MiB.
* `_phase10_vram_probe(label)` is a no-op unless `KILN_PHASE10_VRAM_PROBE`
  is set (so the symbol stays in the binary at all builds, neutralising one
  source of binary-footprint shift between commits).
* Probes inside `checkpointed_forward_backward`:
  * `after_embed` — once, after `model_forward_embed` returns.
  * `boundary_pre_{i}` / `boundary_post_{i}` — around each Step 1 boundary
    forward.
  * `after_boundary_forward` — once, after Step 1 completes.
  * `recompute_pre_{i}` — once per Step 2 segment iteration, at top.
  * `recompute_post_mono_{i}` / `recompute_post_tiled_{i}` — once per Step 2
    segment iteration, after `accumulate_grads` (monolithic) or after
    `total_loss += seg_loss; continue;` (tiled).
  * `after_recompute` — once, just before `let avg_loss = total_loss /
    num_segments`.

Each probe shells `nvidia-smi`, so they cost ~5 ms each, ~150 ms total per
SFT step at seg=8. This is rounding error against the 9.96 s seg=8 step.

### Probe sequence at seg=4 (auto-config) — both commits, byte-identical, both OOM

At the trainer's auto-configured 4 segments × 8 layers (A6000 default
when `KILN_GRAD_CHECKPOINT_SEGMENTS` is unset), both arms hit the GPU rail
during segment-0 recompute and OOM:

| Probe                         | c983ca7 (MiB) | 1edecd7 (MiB) |
|-------------------------------|--------------:|--------------:|
| post-load                     |        16 346 |        16 346 |
| post-warmup                   |        32 778 |        32 778 |
| `after_embed`                 |        32 778 |        32 778 |
| `boundary_pre_0..3`           |        32 778 |        32 778 |
| `boundary_post_0..3`          |        32 778 |        32 778 |

| `recompute_post_mono_0`       |       (OOM)   |       (OOM)   |

The 4×8 segmentation makes per-segment recompute hold ~2× the activations of
8×4, so peak hits the rail and any sub-rail delta between the two commits
is invisible. This is why we re-ran with `KILN_GRAD_CHECKPOINT_SEGMENTS=8` —
not because seg=4 was wrong, but because seg=4 saturates.

### What the seg=8 probe sequence tells us

* Boundary forward (Step 1) on Qwen3.5-4B at T=2048 with `KILN_USE_FLCE=1`
  costs **0 MiB** above post-warmup baseline. All 8 boundary probe pairs
  show 29 738 → 29 738. This is consistent with monolithic boundary forward
  using `current.detach()` between segments and with `LinearAttentionState`
  being scoped per segment so its allocations are freed before the next
  iteration.
* Step 2's first seg-iteration is the entire +13.4 GiB step delta. The
  later 7 seg-iterations grow peak by 64 MiB total
  (`recompute_post_mono_0` 43 114 → `recompute_post_mono_1` 43 178; flat
  thereafter). This 64 MiB is likely a small allocator-pool refill, not an
  activation/grad delta.
* Mechanically, segment 0's recompute cost is dominated by:
  - one tail forward through layers 8..32 (24 layers, FLCE on the head,
    monolithic),
  - one backward through those 24 layers + LM head to produce
    `grad_at_seg_output` for segment 0,
  - one forward+backward through layers 0..8 with LoRA Vars tracked.
  All later segments do the same 8-layer recompute but their tail-forward
  shrinks (fewer layers from `seg_idx + 1`), and the cached allocator pool
  is already sized for the worst case from segment 0. Hence the flatness.
* Both commits' probe sequences agree at every step, including the
  jump magnitudes. **No segment grows by a different amount at 1edecd7
  than at c983ca7.**

## Interpretation

The expected outcome of this audit was to discriminate hypotheses (1) and
(2) from PR #641 by inspecting the per-segment delta. The unexpected
outcome is that the delta is zero at both segmentations on A6000.

The cross-hardware view, comparing PR #641's A40 numbers to this run's
A6000 numbers (both at the trainer's seg=8 configuration):

| | A40 8×4 (PR #641) | A6000 8×4 (this run) |
|---|------------------:|---------------------:|
| c983ca7 post-warmup       |  ~16 600 |   29 738 |
| c983ca7 T=2048 peak       |   23 367 |   43 178 |
| c983ca7 step delta        |   ~6 800 |   13 440 |
| 1edecd7 post-warmup       |  ~16 343 |   29 738 |
| 1edecd7 T=2048 peak       |   42 023 |   43 178 |
| 1edecd7 step delta        |  ~25 700 |   13 440 |
| post-warmup 1edecd7−c983ca7 | ≈ 0    |        0 |
| step delta 1edecd7−c983ca7 |  +18 656 |        0 |

A6000's post-warmup baseline is ~13 GiB higher than A40's at *both*
commits, but the per-step growth on A6000 is identical between commits
while on A40 it differs by 18.6 GiB.

The pattern that lines up with this is a candle-CUDA caching-allocator
sizing decision that depends on (a) detected total VRAM and (b) something
in the executable's binary layout that PR #638 perturbs. On A6000 (49 140
MiB) the allocator already grabs the fatter pool at warmup regardless of
which binary is running. On A40 (46 068 MiB, only 6.2% smaller) the
allocator at c983ca7 sizes a smaller pool; PR #638 nudges the A40 binary
into the same fatter-pool decision A6000 uses by default. The cost of
that decision is paid by allocator-pool residency, not by anything
candle's autograd can see — which explains why the loss is bit-exact.

This is consistent with **hypothesis (1)** from PR #641 — and inconsistent
with **hypothesis (2)** (autograd graph structure shift), because a graph
structure shift would show up as a per-segment delta on A6000 too.
However, this Tier 1+2 round on A6000 cannot conclusively *confirm*
hypothesis (1): we observed that A6000 doesn't show the regression, and
that the A40 baseline is consistent with a smaller pool, but we did not
inspect the candle allocator's pool sizing logic directly.

The Tier 3 work the PR #641 audit recommended ("revert *only* the kernel
files from 1edecd7 to c983ca7 state, keeping forward.rs post-#638") is
also moot on A6000 — both arms already produce identical peak, so
reverting kernel files at this hardware would be an empty A/B.

## Recommended next slice

**Re-target Tier 2 to A40.** The instrumented harness from this PR (the
slim bench + dispatch tracing + per-segment VRAM probes) should be run on
A40 — the GPU where the regression IS visible. Two outcomes:

1. **Probes inserted, regression still visible.** A40 8×4 c983ca7 peak
   stays near 23 GiB; A40 8×4 1edecd7 peak stays near 42 GiB. Then the
   per-segment delta tells us *where* in the seg-iteration loop the +18
   GiB lives — uniform across all 8 segments (hypothesis 1) or
   concentrated in specific ones (hypothesis 2).

2. **Probes inserted, regression vanishes.** Then the binary footprint of
   the Tier 2 patch (additional pure-Rust symbols + a `nvidia-smi`
   `Command` builder) is itself enough to perturb the allocator-pool
   decision. That would be strong indirect evidence for hypothesis (1):
   a binary-footprint change with no runtime semantic effect can
   masquerade as an 18 GiB allocator-pool delta. If this is the
   verdict, the prescribed fix is **not** to revert PR #638 by itself
   but to expose a deterministic candle-CUDA pool-sizing knob (or set
   one via environment/CUDA-allocator tuning) so future PRs aren't
   sensitive to incidental binary changes.

Either outcome converges on the same prescription: **the Phase 10 §1
RMSNorm CustomOp2 path is mathematically correct (bit-exact loss) and
the regression is paid by allocator residency, not by saved-tensor or
graph correctness.** The next Phase 10 §1 task should not block on
finishing Tier 2 — instead, ramp T past 2048 on A40 with the existing
`KILN_GRAD_CHECKPOINT_SEGMENTS` knob and verify the saved-tensor
reduction PR #638 was meant to deliver actually shows up at T≥4096 (the
regime PR #638 targeted), even if the +18 GiB allocator-pool penalty is
unavoidable at T=2048.

Eric's call on whether the +18 GiB peak at T=2048 on A40 is severe
enough to revert PR #638. Two facts that bear on the decision:

* Final training loss is bit-exact between c983ca7 and 1edecd7 at both
  hardware configurations, so the kernel itself is correct.
* On A6000 the regression is invisible — Phase 10 §1 RMSNorm work
  proceeds normally on A6000 with no extra cost.

## Bench harness

The slim bench used at both commits and the patcher script are
reproduced for reproducibility:

* The slim single-cell bench source is reproduced in
  `scripts/phase10_apply_patches.py` (which writes the bench file into
  place when run). The 6-cell `crates/kiln-server/examples/phase10_rmsnorm_bench.rs`
  on main is intentionally **not** modified by this PR — the slim bench
  is generated on demand from the patcher script so the e2e validation
  harness on main is preserved.
* `scripts/phase10_apply_patches.py` — applies the Tier 1 dispatch
  tracing patch (handles the c983ca7 single-env-var variant and the
  1edecd7 two-env-var variant), the Tier 2 VRAM probes, and writes the
  slim bench into place. The script is idempotent: running it twice on
  the same checkout is a no-op.
